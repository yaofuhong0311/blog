> 本文是「沙箱底层机制」系列第 2 篇。[第 1 篇（选型总览）](/posts/sandbox-deep-dive/)讲了七个角度和隔离阶梯，这篇往下钻一层：把"启动"和"密度"这两个角度背后的机制拆开看。

上一篇说过，每多一层隔离就要交一次"隔离税"——容器共享宿主内核不用交，microVM 每个实例都得自己 boot 一份 guest 内核加 PID 1，约 100ms 起步。snapshot、UFFD、CoW 这些听起来各自独立的技术，其实是同一件事：**把这笔税摊薄、懒付、共享着付**。

而它们底下压着同一个硬件机制：**缺页异常（page fault）**。这个本来用于报告"内存访问出了问题"的错误信号，被工程师们一次又一次地挪用来做正经事。这篇就顺着它讲。

## 地基：可写位与缺页

CPU 的 MMU 在每次内存访问时都会查页表项（PTE），其中有一个"可写位"：为 1 放行写入，为 0 时写操作触发缺页异常，陷入内核。**后文的全部机制都围绕这个位展开**——内核可以故意把可写位置 0，让"写"这个动作变成一次可被拦截的事件；也可以故意不建立映射，让"访问"本身变成一次可被拦截的事件。

缺页从"错误"变成"钩子"，后面的一切由此展开。

## 第一次挪用：fork 与 CoW

fork 复制一个进程，如果真的把内存逐页拷一份，又慢又浪费——绝大多数页 fork 完马上 exec 就扔了。因此内核采用了延迟复制的策略：

```text
fork 前：物理页 P，refcount=1，父进程 PTE 可写位=1
fork 后：父子都指向 P，双方 PTE 可写位都置 0，refcount=2
某方写 P：可写位=0 → 缺页 → 内核分配新页 Q 拷贝 → 写方指向 Q、恢复可写；P 的 refcount 2→1
P 只剩一方指向时：下次写直接改回可写位，免拷贝原地写
```

两个容易混的数：**可写位是每个页表项的硬件位，refcount 是每个物理页的计数**，不是一回事。还有一个容易想歪的点：CoW 是**对称**的——fork 时父子的可写位一起置 0，谁先写、谁复制出私有副本，不存在"永远拷子进程那份"。顺带一提，那个"refcount 降回 1 就免拷"的优化，其竞态窗口就是著名的 Dirty COW（CVE-2016-5195）的成因——性能优化节省的开销，往往在别处以复杂度和风险的形式偿还。

对沙箱的直接推论：**fork 出 N 个沙箱，内存靠 CoW 自动分家，但磁盘默认是同一个**——文件系统不归 fork 管，得额外叠一层磁盘 CoW（OverlayFS 或块级 reflink）才能互不干扰。

## 启动阶梯：snapshot 省的是 boot，不是分配

先把启动快慢排个序：

```text
容器 fork/exec ─► microVM 冷启 ─► snapshot 恢复 ─► +UFFD ─► +CoW/暖池
  ~几十ms          ~125ms+         ~几十ms       working-set   密度
 (不 boot 内核)  (boot 独立内核)   (跳过 boot)     决定延迟     (只读页共享)
```

判定快慢的第一性问题只有一个：**这条路径要不要从零 boot 一个内核、拉起 PID 1**。容器不用（clone/unshare 建 namespace → cgroup → pivot_root → seccomp → execve，全程借宿主内核）；microVM 必须（这就是那 100 多毫秒的隔离税）；snapshot 恢复则把"boot 完成后的状态"直接存了下来——从 state 文件恢复 vCPU 寄存器、KVM 状态、设备状态然后 resume，**整段 boot/init 被删掉了**。

顺带纠正一个常见误读：Firecracker 官方的"125ms"是 best-case 上界，强依赖裁剪过的 guest 镜像；它快的真因是"极简设备模型 + 未压缩内核直启 + 裁剪 guest"，不是单纯省了 BIOS。

snapshot 落到磁盘上是三件东西：**mem 文件**（guest RAM 的 raw dump，增量快照只存脏页）、**state 文件**（vCPU/KVM/设备状态，"立刻全量恢复的骨架"）、以及**磁盘 rootfs——它不在快照里，用户自管**，快照也不会自动 flush 磁盘，一致性是你的责任。很多"恢复后磁盘状态异常"的问题源于此。

## 第二次挪用：恢复时，内存怎么来

state 文件很小，麻烦的是 mem 文件——guest 的几个 G 内存总不能恢复时全量读回来。答案还是缺页：**故意不加载，等 guest 真的访问到哪页，缺页信号来了再补哪页**。同一个思路有两条实现路径，密度收益差一档：

- **File backend**：对 mem 文件做 MAP_PRIVATE 映射，宿主内核自己按缺页懒分页。关键在于 N 个沙箱映射同一个 base 快照时，**没写过的页在宿主机 page cache 里就是同一份物理内存**——内核代码、libc、解释器、模型权重这些大块只读内容全程单副本共享。这是高密度部署真正的来源。
- **UFFD（userfaultfd）**：把缺页处理从内核搬到**用户态**——guest 首次触到未填充页，内核把事件投递给一个用户态 handler 进程，由它决定从何处获取这一页的数据（本地文件、去重存储、远端对象存储都行），填进去再唤醒。

UFFD 有个普遍的误读：它的卖点**不是"懒"**——File backend 本来就是懒的。它的差异化是"**缺页处理权交给用户态**"带来的页源可插拔。代价也直白：每个冷页多一次绕道用户态的延迟，handler 崩了整个 VM 挂起，而且页是 copy 进各 VM 的**私有匿名页**——共享的只有磁盘文件的 page cache，密度收益比 File backend 弱一档。

同一招还有第三个变体：**EPT violation 也能被挪用做懒加载**。它发生在 guest 物理地址到宿主机物理地址的翻译层（和 UFFD 所在的宿主机自身内存管理是两层），快照恢复时故意让 EPT 表大面积不映射，guest 一碰就陷出，VMM 不当错误处理，取来页内容、补上映射、继续跑。两层机制在完整链路里可以协同工作。

## 密度经济学：血缘共享、内容去重和一条生产验证

共享内存有两条完全不同的路：

- **靠血缘**：fork、File backend 这类——因为映射自同一个源，天然就是同一份物理页，零成本。
- **靠扫描**：KSM——后台持续扫描全系统内存，发现两页**内容**逐字节相同就合并成一份（谁写谁再分裂）。它不需要任何血缘关系，代价是持续的 CPU 开销，而且**跨租户去重构成侧信道**，只该在同信任域内开。

"共享 guest 内存"这一说法的物理本质值得澄清：guest 内存从来不是独立存在的东西，它一直是宿主机物理内存的一个翻译视角——所谓 N 个 VM 共享，就是它们各自的 EPT 映射表被配置成指向宿主机同一块物理地址。

这套经济学有生产级验证：AWS Lambda SnapStart。init 只跑一次，取快照，之后每个新环境从缓存快照 resume（Java 冷启最多快约 10 倍）。它的工程细节恰好印证了前面所有要点：512KB chunk 化 + 三级缓存 + working-set 主动预取——因为朴素的 demand-loading 走网络存储会把延迟收益全吃回去。核心的成本模型一句话：**snapshot 把 init 成本从 O(N) 变成 O(1)，N 个沙箱摊销一次初始化**。

## 同一形状在磁盘层重放：qcow2

最后看一眼磁盘，你会发现同一个机制以另一种形式再次出现。QEMU 的 qcow2 格式把虚拟磁盘切成 64KB 的簇，用**簇表**记录"虚拟磁盘第几簇实际存在文件哪个位置"——这和页表把虚拟地址翻译成物理地址是**同构**的。backing file 则是簇表查不到时的回退数据源：读 overlay 未命中就穿透读 base；写入时才分配新簇、合并写入、更新簇表——和 fork/CoW 的"共享只读源、首次写复制到私有位置、更新映射"是同一个形状，只是从内存页表层搬到了磁盘簇表层。

这里有一个值得精确区分的概念：qcow2 的簇分配**只由写触发**，纯读取穿透 backing file 不会在 overlay 里留下任何东西——所以它是"写时复制"，不是"懒加载"（懒加载是任意首次访问、包括读、就触发加载）。两者同属"按需跳过不必要工作"家族，但触发条件不同，混用名称会在排查问题时造成概念混乱。

## 收尾：一张机制归属表

| 机制 | 骑的硬件位 | 出现在哪 | 干什么 |
|------|-----------|---------|--------|
| 脏页追踪 | dirty bit | 快照的"拍"侧 | 挑出改动页，增量快照只存脏页 |
| CoW | writable bit | fork、快照的"恢复"侧、OverlayFS、qcow2 | 共享只读源，写时复制 |
| 懒加载 | 缺页/EPT violation | File backend、UFFD、EPT 挪用 | 访问到才加载，恢复延迟随 working-set |

用 mem 文件串起来记：**脏页管"写入快照文件时只挑改动"（入口），CoW 管"读快照回来时共享、写时复制"（出口）**。一个错误信号，四次挪用（fork 的 CoW、快照懒加载、Wasm 的边界检查、EPT 的按需映射），撑起了 microVM 世界的启动速度和部署密度——这大概是我这轮调研里最喜欢的一条暗线：**优秀的系统设计常常不是发明新机制，而是给旧信号找到新用途**。
