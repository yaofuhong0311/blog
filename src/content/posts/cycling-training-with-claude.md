我有晨间骑行台训练的习惯，随之而来的是数据散落问题：体重在小米的 App 里，骑行功率数据经 Zwift 上传到 Garmin，训练日历的复盘在 intervals.icu。三个数据源互不相通，每次想回答"最近状态怎么样、该加量还是该休息"，都要挨个打开对照着看。

这篇记录我怎么用一个自写的小工具加一个开源 MCP 服务，把这条链打通，最后让训练分析变成对 Claude 说一句话的事。

## 整条链路

![训练数据链路](/images/posts/training-data-pipeline.svg)

链路分两半。左半段解决**数据进得来**：搭好之后全自动运行；右半段解决**数据看得懂**：这是 MCP 发挥作用的地方。

## 左半段：体重同步的小工具与一个坑

骑行圈常说的功体比（W/kg）需要体重数据持续在线，但小米秤的数据出不了小米生态。于是写了 [xiaomi2weight](/tools/xiaomi2weight/)：抓取体脂秤数据，写入 Garmin Connect，再由 Garmin 自动同步到 intervals.icu——体重从此不需要手动记录。

这条链上踩过一个值得记录的坑：intervals.icu 上的体重曾长期显示为一个固定值，怎么同步都不变。排查后发现根源不在同步链，而在 **Strava 的个人资料**——icu 同时连接了 Strava，而 Strava 资料页里手填过一次体重，每次活动同步时这个固定值会把秤的真实读数覆盖掉。数据链路的问题不一定出在链路本身，也可能出在某个想不到的旁路数据源。

另一个实践经验：称重时机对读数的影响能达到一公斤级别，固定在每天同一时间（我是晨起后）称，趋势才有意义。

## 右半段：garmin_mcp 接入 Claude

Garmin Connect 里其实什么都有——活动、FTP、VO2max、训练负载、HRV、睡眠——缺的是一个好用的分析入口。开源项目 [garmin_mcp](https://github.com/Taxuspt/garmin_mcp) 把 python-garminconnect 封装成了 MCP 服务，暴露 110 多个工具，挂进 Claude Code 之后，模型可以按需调用任意组合。

配置只有几行（`settings.json` 里注册 MCP 服务，凭据走环境变量），之后的使用方式就是自然语言。比如问"我最近训练状态怎么样"，Claude 会自己决定调哪些工具，然后给出跨数据源的综合回答。以下是一次真实分析（2026 年 7 月中旬）：

> **状态判定**：Garmin 给出的训练状态是 PEAKING（竞技状态上升期）。
> **体能指标**：骑行 VO2max 从月初的 66 升到 68；FTP 于 7 月 5 日更新为 300W。
> **负载结构**：急性负载 825，慢性负载 1094，负载比 0.7——处于偏低区间，意味着近一周的量低于身体已适应的水平，有加量空间，不必担心过度训练。
> **训练平衡提示**：低强度有氧占比偏高，若目标是继续提升 FTP，可以适当增加阈值区间的课次。

这段分析人工做也能做，但需要在 Garmin 的三四个页面之间跳转、自己记住各指标的含义与合理区间。交给模型之后，它调了三个工具、二十秒内给出结论——**MCP 的价值不在于取到数据，而在于取数、解读、跨源关联这三步被压缩成了一次对话**。

## 几点体会

- **管道和分析应当分开建**。左半段（同步链）是确定性任务，用普通脚本一次搭好就不用管；右半段（分析）是开放性任务，适合交给模型。用模型去做定时同步是浪费，用脚本去做综合分析是为难自己。
- **MCP 工具的粒度决定了分析的自由度**。garmin_mcp 把接口拆得很细（单独的 FTP 工具、负载趋势工具、HRV 趋势工具），模型可以按问题自由组合；如果只暴露一个"导出全部数据"的粗接口，每次分析都要吞吐大量无关数据。这与给 Agent 设计任何工具集的原则一致。
- **数据的可信度要先于分析**。Strava 覆盖体重那个问题存在期间，任何基于功体比的分析都是错的。接入模型分析之前，值得先把数据链本身核对一遍。

工具链本身没有任何复杂技术，两个组件一个是自写的小脚本、一个是别人开源的 MCP 封装。值得写下来的是这个组合方式：**自动化解决数据流动，模型解决数据理解**——这个分工模式应该能推广到训练数据之外的很多场景。
