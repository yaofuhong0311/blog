> 背景：一个常驻的多租户 Agent 服务里，"技能"（skill）是平台资产——预置的、用户让 agent 优化出来的、算法进化出来的，都算。策略是"**用可以，拿走不行**"：平台内可以自由使用技能完成工作，但技能本身的说明内容不能被导出带走、到别处复现。这篇是把这套防导出体系从零做出来的设计总结与踩坑记录。

先把最反直觉、也是全篇地基的一句话放前面：**模型读得到的内容，防不住语义级泄露。** 技能说明必须进模型上下文才能被使用，一旦进了上下文，用户就能让模型"换个说法讲出来"。这是业界公认的天花板（对应 OWASP 大模型风险清单里的敏感信息泄露一项）。所以下面这套东西从设计之初就不追求"根治"，只追求**抬高导出成本**——拦住顺手就想搬走的大多数，让真正用心的少数付出不成比例的代价。

## 三条核心认知

动手写任何代码之前，先有三个判断决定了整套架构的形状。

**第一，要保护的是"能力"，不是"文字"。** 一段被改写的说明文字，单独拿走没有用；只有"说明 + 能跑的执行核心"都拿到，技能才能在别处复现。所以终极方案是把执行核心留在服务端（skill-as-a-service），文字层面的防线只是过渡手段。想清楚这一点，才不会在"文字防线"上过度投入。

**第二，天花板是语义级泄露，任何字面方案都只是抬成本。** 承接开头那句——指纹、相似度这类手段都属于"抬高成本"，不属于"根治"。认清这条，才会把力气花在性价比最高的拦截点上，而不是追求一个不可能达到的"滴水不漏"。

**第三，判别信号是"行为模式"，不是"单点相似"。** 一次完整的改写导出，会**逐段覆盖**技能正文；而正常完成任务产出的成果，只会在个别段落涉及几个领域词。所以"多段命中"远比"单段高相似"可靠——这是后面所有阈值设计的核心依据。

有个认知是踩坑换来的：最初想做成"作者主动打标保护"（opt-in），很快发现这是伪保护——**想导出的人不会给自己上锁**。正确做法是反过来：默认全量保护，只有显式声明放弃保护的技能才不管。默认值的方向，直接决定了一套保护机制是真有用还是纯摆设。

## 分层架构

![技能防导出的分层防御](/images/posts/skill-protection-layers.svg)

整套体系的关键设计，是把"通道"和"层"分开看：**在哪个出口拦、用哪种方法查，是两件正交的事**。

出口通道分三种，处置方式截然不同：

- **聊天回复**：永不拦，只记日志。这是最重要的一条纪律——拦了回复，技能对用户就形同报废，因为用户正常问技能相关的问题时，回复里天然就会引用技能片段。**防导出不等于防使用。**
- **写文件**：命中则不落盘，返回一段策略说明。
- **下载**：命中则拒绝，响应体里带同样的策略说明。

检测方法分两层，抓的是不同的搬运手法。

### 指纹层：抓逐字搬运

按**行**切，归一化处理（转小写、折叠空白），太短的行丢弃（短行在不同技能间容易撞车）。每行算 sha256，建一张"哈希 → 技能名"的倒排索引。出口文本同样切行查表，同一技能命中三行以上即判定为搬运。

特点是**零误报**——哈希相等意味着逐字相同，不可能误伤。代价是改一个字就失效。它专治"整段复制粘贴"这种最省事的导出方式。

### 语义层：抓换个说法的改写

指纹层挡不住"把说明重新表述一遍再导出"，这正是语义层要弥补的短板。

这里用了 TF-IDF 而不是神经网络 embedding。**TF-IDF 是一种统计向量**：TF 是某个片段在本文里出现的频率，IDF 是它在整个语料里的稀有度，两段文本的余弦相似度，衡量的是"它们共享了多少高辨识度的字符片段"。

它和 embedding 的差距要讲清楚：**TF-IDF 只认字面、不懂含义**——"合并 PDF"和"把多份 PDF 拼成一份"在它眼里几乎不相似；embedding 是神经网络把语义压成向量，同义改写甚至跨语言翻译都能对上。一句话概括：TF-IDF 是查重，embedding 是查意；查重能拦住大多数怕麻烦的人，查意才拦得住少数真正用心改写的人。

那为什么第一版选了较弱的 TF-IDF？纯粹是工程现实：内部网关当时没有 embedding 的调用路由（实测直接 404），本地引入一个神经模型要额外增加两百多兆的镜像依赖，而 TF-IDF 所依赖的统计库本来就在镜像里。**零新增依赖、当日就能上线**，同时把向量化那一层留成可插拔接口——日后网关支持了 embedding，只需替换这一层，其余不动。先用能上线的方案把闭环跑通，再谈升级。

一个可能让人意外的结果：**换说法的改写，语义层照样抓得住。** 原因是改写的人很难把术语也换干净——库名、函数名、专有名词一旦替换，说明就不成立了，留着就会被字符 n-gram 抓到。实测中，把一份英文技能说明改写成中文导出，依然命中了三段。

拦截采用**双阈值**：较低的一档作为"影子线"，只发事件、不拦截，用来收集真实流量的相似度分布；较高的一档才是"拦截线"，留出误报余量。两条线各司其职——影子线负责观测，拦截线负责保守出手。

### 阈值是怎么定出来的

阈值不是拍脑袋，是标定出来的。造三类合成样本做对比：整篇改写、正常任务成果（用了技能但产出的是数据本身）、完全无关的文本。实测下来三类的相似度分布拉得很开——改写命中的段落显著高，正常成果很低，无关文本几乎为零。影子线落在"改写全命中、成果零误报"的位置，拦截线再往上抬一档留余量。

这里有个可复用的方法论：**影子模式先行**。任何拦截规则上线前，先让它只发事件、不真拦，收一段真实流量校准分布，确认误报可控了再切成真拦截。

## 踩过的坑

1. **主动打标是伪保护**：让用户自己给技能打保护标记，等于没保护。默认全量、显式退出。
2. **系统预置技能是盲区**：审计范围最初只扫了 agent 包内的技能，把平台最核心的一批系统预置技能整个漏掉了——是用真实导出场景实测出"零事件"才发现的。教训：**上线后必须拿真实场景实打一遍，不要相信"应该覆盖到了"**。
3. **回复通道绝不能拦**：拦了回复，技能对用户就是坏的。再强调一次，防导出不等于防使用。
4. **拦截要返回"策略说明"，不是报错**：模型收到一段裸报错，会重试、会道歉、会绕；收到一段清晰的策略说明，会如实转告用户"这个内容受保护、不能导出，但可以正常用它完成任务"。给模型看的错误信息，本身就是一种接口设计。
5. **fail-open 是铁律**：检测代码自身抛任何异常，一律放行。防护逻辑挂掉，绝不能把正常功能一起拖垮。
6. **kill switch 走配置、不走发版**：所有开关每次调用现读配置，改一下即时生效。一旦出现误伤，能秒级关闭，不必等一次发版。
7. **选型前先探基础设施**：真正动手做 embedding 方案之前，先实测网关到底有没有那个路由（结果是 404）。省掉了"做到一半发现依赖根本不存在"的返工。

## 已知天花板与升级路线

诚实地列出**当前防不住的**（按难度递增）：连术语一起替换的深度洗稿、整篇翻译成另一种语言导出、分很多次每次搬一小段的"蚂蚁搬家"，以及最根本的一条——纯散文类技能的语义级泄露，**没有架构层面的解**，因为模型读得到，用户就问得出。

升级路线按性价比排序：

1. **会话级关联信号**：把"读审计"和"出口相似度"两个已有信号联合判定——"刚读过这个技能 + 出口内容中等相似"才告警。两边的事件都已经存在，几乎零成本地降误报、提召回。
2. **模糊哈希（SimHash）**：内容越像、指纹的比特差异越小，用来弥补"只改了几个词"这个空档——此时精确哈希已失效、段级 TF-IDF 又可能不够敏感，同样零依赖。
3. **网关接入 embedding**：真正对付深度洗稿和翻译导出，代码侧的可插拔接口已经留好，只等基础设施跟上。
4. **skill-as-a-service（终极方案）**：把有算法或脚本核心的技能，其核心逻辑下沉成服务端可调用的工具，模型只能调用、拿不到实现——这才是**真的拿不走**。纯散文类技能无核心可下沉，只能靠前面几层加自律兜底，接受天花板。

而最有价值的一条，是**改技能的生产规范**：让技能从诞生起就长成"薄说明 + 服务端可调用核心"的形态。这样每一个新技能天生就是"拿不走"的——文档公开也无所谓，因为引擎跑在你的服务器上。这和[沙箱系列](/posts/agent-runtime-sandbox/)里反复出现的判断是同一个：**真正的护城河不在能被复制的那部分，而在跑在你这边、复制不走的那部分。**
