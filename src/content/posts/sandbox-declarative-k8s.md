> 本文是「沙箱底层机制」系列第 4 篇。前三篇从隔离、缺页、逃逸的角度讲了沙箱"是什么、怎么快、怎么被打穿"，都属于单个实例内部的机制。这一篇转向管理侧：一群沙箱在 K8s 上如何被编排。这正是第一篇提到的"两条主线"中的第二条——从"改一次"走向"持续对账"。

在 K8s 上跑沙箱，第一反应往往是命令式的：应用代码直接调 API 建一个 Pod，发出去、拿到响应，成了就是成了。这套做法有一个根本缺陷——**之后这个 Pod 出了什么事，没有任何机制会自动发现并处理**。节点维护、被驱逐、OOM、意外删除，除非你自己写了一整套轮询和重试逻辑去盯着，否则它消失了就是消失了。

声明式模型换了个思路：不再"下一道命令"，而是声明一件事应该存在，然后交给一个持续运行的循环去兜底。理解这个转变，是理解 K8s 全部编排机制的钥匙。

## 声明式的核心：reconcile loop

声明式的做法是这样的：你不说"去建一个 Pod"，而是声明"我要一个具备这些属性的 Sandbox 资源存在"，把这份声明写进 K8s 的存储。然后一个叫 **controller** 的组件在后台持续运行，它唯一的工作是不断检查现实与声明的期望是否一致，不一致就补救。

这个检查—修复的循环叫 **reconcile loop**，它有两个容易被误解的特性：

- **它贯穿资源的整个生命周期，不绑定在"创建那一刻"。** 只要资源还存在，任何相关变化都会触发它重新运行——底层 Pod 崩了会触发，几小时后用户改了期望配置会触发，周期性重新同步到点也会触发。它不是"创建时看一眼就完事"。
- **它比对的维度不局限于某一种。** reconcile 比对的是期望状态与实际状态在**任何维度**上的差异——镜像对不对、资源限制对不对、Pod 是不是消失了需要重建，都可以编进 reconcile 逻辑。它不是天生只做某一种检查。

它真正解决的问题是"漂移了没人管"。直接建 Pod，之后因任何与用户请求无关的原因意外消失，命令式代码不会自动重建；而 reconcile loop 会自己发现底层 Pod 不见了并补上，覆盖的是**所有**可能导致偏离期望状态的原因，不是提前想到的那几种。一句话概括：命令式是"我让它发生一次"，声明式是"我声明它该长什么样，剩下的、包括意外发生后怎么补救，交给一个持续运行的循环兜底"。

实现上，controller 并不傻等或轮询。它维护一个工作队列，被监听的资源发生任何变化都会被塞进队列；同时通过一套叫 **Informer** 的长连接 watch 机制被动接收变更通知，并在本地缓存一份最新状态，避免每次都打网络请求。

## CRD 是名词，Controller 是动词

要让 K8s 理解一个自定义的 `Sandbox` 类型，需要两样东西配合。

**CRD（CustomResourceDefinition）是名词。** K8s API server 默认只认识内置类型（Pod、Deployment、Service 等），这些写死在它的代码里。CRD 是提交给 K8s 的一份"新词定义"——告诉 API server 存在一种叫 `Sandbox` 的新类型，规定它能有哪些字段（`spec.image`、`spec.isolationLevel`、`spec.resources.cpu`）、每个字段什么类型、什么校验规则。本质是一份 schema。这些字段的意义在于，它们是**调用方声明"要什么"与 controller"该怎么造"之间的契约**——controller 要读这些字段，才知道具体该建一个什么样的 Pod。CRD 注册后，API server 会自动获得存取这种新类型对象的能力，无需编写任何服务端存储代码。

**但 CRD 本身不会让任何事发生。** 注册好类型、建了一个具体的 `Sandbox` 对象，并不会自动创建出对应的 Pod——这个对象只是安静地躺在存储里，是一份纯数据。真正让"这里应该有个 Sandbox"变成"真的建出一个 Pod"的，是你写的 **controller**：它盯着这类对象，发现声明存在但实际没建出来，才动手去建。CRD 是名词，controller 是动词，两者配合才构成完整的自定义资源体系。

顺带厘清一个底层事实：K8s 所有状态（每个 Pod、每个 Deployment、每个自定义对象）本质都只是 **etcd** 里的键值记录。etcd 是一个通用的分布式强一致键值库，并非 K8s 专属发明，它完全不懂"Pod"这些概念；所有 K8s 特有的语义（字段校验、权限）都在 **API server** 这层完成。API server 是唯一直接对话 etcd 的组件——kubectl、controller、kubelet 都只跟 API server 打交道。

## 为什么建 Pod 不用自己写 controller

这里有个常见的困惑：既然 controller 这么重要，为什么我直接建一个 Pod 也能跑起来，从没写过 controller？

答案是——**你用的是别人写好的 controller**。Pod 是 K8s 内置类型，K8s 自己已经写好了管它的两个 controller：`kube-scheduler`（监听未调度的 Pod，为它分配节点）和 `kubelet`（跑在每个节点上，监听分配给本节点的 Pod，真正调用容器运行时把它跑起来）。这两个是"内置版"controller，与你自定义的 Sandbox controller 用的是同一套模式——声明、监听、对账、执行——只是 K8s 工程师早已写好、随集群自带。**只有当你发明一个全新类型（比如 Sandbox）时，才需要自己写 controller，因为没人替你写过。**

所以一个 Pod 的启动，本质是两步：先把对象存进 etcd（经 API server，声明期望状态），再由 controller（内置的或自定义的）把这个期望状态变成真实运行的东西。第一步"只是存数据"这件事，因为紧跟着有内置 controller 接手、跑得又快又稳，让人几乎感觉不到，才容易忽略中间这个环节的存在。

## 一个应用：超时前先快照再终止

把声明式模型和前几篇的机制串起来，可以设计一个实用的行为：controller 监听到某个沙箱 Pod 即将超时，**在真正终止它之前**先触发一次快照（第二篇讲的 mem/state/disk 三件套），把内存与磁盘的完整状态捕获成文件，**然后**再安全终止。

**状态不会丢，关键在顺序。** 只要是"先快照、再终止"，状态在终止前已被完整复制进快照文件；杀掉进程只是丢了"这个进程本身"，它承载的状态早已搬进快照。日后恢复是拿快照造一个新实例、把状态还原进去，不是原进程复活，但对使用者效果等价，因为状态是连续的。这正是第二篇提到的 AWS Lambda SnapStart 在生产环境验证过的模式。

需要说明一个前提：这套完整状态快照能力，目前只有 microVM（Firecracker/Kata）是成熟、生产可用的路径。普通容器有对应的尝试——CRIU（Checkpoint/Restore In Userspace），能把 Linux 进程的完整状态冻结导出、日后还原——但尚不够成熟。**根源仍在共享内核**：普通进程的状态与宿主机内核里全局共享的数据结构（路由表、网络连接追踪表、PID 编号池、共享内存段）纠缠在一起，而这些结构同时被其他进程使用，想干净地"只抠出这一个进程的那部分状态"天然更难。microVM 的 guest 有独立内核，整个状态自成一体、不与宿主机任何东西纠缠，这才是它快照能做得干净可靠的根本原因。CRIU 不是做不到，是很难做到 100% 完整可靠。

## 同一个模式的另一个投影：Service

理解了 reconcile loop，你会在 K8s 里到处看到它的影子。**Service** 就是一个好例子。

Pod 每次重建，IP 都会变，不能硬编码。Service 提供一个稳定不变的虚拟 IP 和 DNS 名字，背后实际对应的 Pod 换了、IP 变了，这个"门牌号"不变。而维持这个门牌号有效的，正是一组持续对账的 controller：**EndpointSlice 控制器**持续 watch 哪些 Pod 符合 Service 的筛选条件且就绪，把答案写进一张表；**CoreDNS** watch 这张表，维护"名字→地址"的映射；**kube-proxy** 同样 watch 这张表，负责把流量导向真实 Pod。每一环都是"声明期望—持续对账—修正现实"这同一个模式的复现。

不过对沙箱场景要留意一个假设错配：普通 Service 假设背后是一堆可互换的副本，随便转发给一个都行。但沙箱要连的往往是**某个会话专属的那一个实例**，不能随便转发（会导致跨会话数据串扰）。常见解法有三种：给每个沙箱配一个专属 Service、直接查 API server 要该 Pod 的当前 IP、或用 headless service 让 DNS 直接返回真实 Pod IP。选哪种，取决于你的寻址是"负载均衡"还是"精确定位"。
