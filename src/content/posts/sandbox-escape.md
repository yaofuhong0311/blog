> 本文是「沙箱底层机制」系列第 3 篇。[第 1 篇](/posts/sandbox-deep-dive/)把隔离强度列为选型的第一个角度，[第 2 篇](/posts/sandbox-page-fault/)讲了缺页机制。这一篇从反面切入：隔离**失效**时，攻击是怎么一步步走通的。

第一篇讲隔离时，我用"离宿主机内核还有几层"来衡量强度。但只看防御方的分层是不够的——真正理解一个隔离机制的边界，要看它在什么情况下会失效。容器逃逸的真实案例可以归纳为三类根源，一类比一类底层，也一类比一类难防：**配置性逃逸（权限给多了）→ 运行时组件漏洞（共用工具的 bug）→ 内核漏洞（共用内核本身的 bug）**。

## 第一类：配置性逃逸

这一类的共同特点是：没有利用任何漏洞。容器被配置成拥有了不该有的权限，容器内进程拿着这份权限，直接、合法地完成逃逸。用一个比喻，这不是破门而入，而是钥匙本来就插在门缝里。常见形态有五种：

- **挂载 docker.sock**：容器内进程通过这个 socket 直接向宿主机 Docker daemon 发 API 请求。注意关键——它不是 exec 进已有容器，而是借宿主机 daemon（root 权限）之手创建一个**全新的、配置任意的容器**（例如 `docker run -v /:/host --privileged`），从而获得对宿主机文件系统的完整访问。常见于 CI/CD 镜像构建场景。
- **`--privileged` 标志**：一次性放开几乎所有限制——全量 capability、关闭 seccomp、关闭 AppArmor/SELinux、开放几乎所有设备节点。拿到后甚至不需要任何漏洞，直接挂载宿主机磁盘设备节点即可读写宿主机文件系统。
- **单独授予危险 capability**：`CAP_SYS_ADMIN`（可执行 mount 等特权操作）、`CAP_SYS_MODULE`（可加载内核模块，等于直接写内核态代码）、`CAP_DAC_READ_SEARCH`（绕过文件权限检查，经典逃逸演示利用它配合 `open_by_handle_at()`）。
- **挂载宿主机敏感目录**：直接挂 `/`、`/etc` 或 SSH 密钥目录，无需任何技巧，改写宿主机的 crontab、systemd unit 或 authorized_keys 即可等待触发执行。
- **共享宿主机命名空间**（hostPID/hostNetwork/hostIPC）：hostPID 让容器可见宿主机全部进程，并可通过 `/proc/<host-pid>/root` 借用其文件系统视角。这几项是节点级基础设施组件的合理需求，普通应用负载一旦出现即为可疑信号。

这一类风险的核心不在"指令合不合法"，而在"谁有资格发出这条指令"。回到第一篇的观点：容器隔离本就不是硬件级别不可逾越的墙，而是一堆可以被单独关闭的软件限制。Docker daemon 执行"创建一个新容器并赋予最高权限"这条指令，是它的正常职责，每天都有合法调用方在用；问题出在挂载 docker.sock、开启 `--privileged`，本质是把"能发出这条指令"的资格错误地交给了一个本应被视为不可信的沙箱负载。**防御的着力点应该是"谁能碰到这个通道"，而不是"这条指令该不该存在"。**

### 一个配套的缓解手段：user namespace 重映射

针对"挂载敏感目录"这类风险，关键缓解点是 user namespace 重映射：开启后，容器内即使显示为 UID 0（root），内核也会维护一张映射表，把它对应到宿主机上一个无特权的 UID（如 100000）。容器进程访问边界外资源时，内核用映射后的无特权 UID 做权限检查，从根源上堵住"root 无视文件权限位"这条路。需要强调的是，**这不是默认行为**，必须专门开启；未开启时容器 UID 0 就是宿主机 UID 0，`chmod` 收紧的权限位对它形同虚设。NFS 协议服务端的 `root_squash` 是同一防护思路在协议层的实现。

## 第二类：运行时组件漏洞

这一类不依赖权限配置。即使容器以非 root 运行、没有危险 capability、没有敏感挂载，依然可能被打穿——因为漏洞出在**容器与宿主机共用的工具本身**。最经典的案例是 CVE-2019-5736。

runc 是 Docker/containerd 底层真正创建和进入容器的工具，运行在宿主机上。执行 `docker exec` 向目标容器注入新进程时，runc 需要通过 `/proc/self/exe`（永远指向当前正在运行的可执行文件）重新执行自身来完成 namespace 切换。攻击链分四步：

1. 恶意容器把自身某个可执行文件（如 `/bin/bash`）替换成精心构造的程序；
2. 有人对该容器执行 `docker exec`，runc 被调起；
3. 由于一处**时序竞争**，容器内的恶意程序在 runc 完成 exec 切换前的极短窗口内，能拿到一个指向宿主机磁盘上 runc 二进制本身、且可写的文件描述符；
4. 恶意程序趁机将宿主机上的 runc 文件整体覆写为攻击者代码。

此后，任何人在该宿主机上调用 runc（哪怕只是启动一个完全无关的新容器），执行的都是攻击者代码，并以 runc 的宿主机 root 权限运行。攻击者由此拿下整台宿主机，不再局限于原容器。修复方式是让 runc 改用 `memfd`（无文件系统路径的匿名内存文件）装载自身副本，堵死这个可被替换的窗口。

2024 年的 Leaky Vessels（CVE-2024-21626）同属此类：漏洞出在 runc 处理工作目录的过程中，导致宿主机上一个本不该暴露的文件描述符泄露进容器。性质与 CVE-2019-5736 一致——漏洞在共用组件本身，与容器配置无关。

值得指出的是，CVE-2019-5736 与第一篇提到的 DNS rebinding 同属 **TOCTOU（Time-Of-Check-Time-Of-Use）** 这一漏洞大类：程序假设"检查完成后状态不会改变"，但检查与真正使用之间存在一个窗口，攻击者专门在这个窗口内篡改状态。前者篡改的是 runc 即将执行的文件内容，后者篡改的是域名解析结果，攻击目标不同，漏洞形状相同。

## 第三类：内核漏洞

这是三类根源中最底层的一类。它不靠配置，也不靠共用工具，而是直接利用内核管理内存与页缓存的代码本身的 bug。这层代码全机器只有一份，容器、宿主机、所有进程共用，没有任何 capability 或 namespace 能挡住它。

- **Dirty COW（CVE-2016-5195）**：利用私有只读映射的 CoW 判断逻辑中的一处时序竞争。一个线程反复对映射区域调用 `madvise(MADV_DONTNEED)`，另一个线程同时通过 `/proc/self/mem` 写同一地址；竞争窗口内内核判断出错，把写入数据直接落进真实的页缓存，而非按规则生成私有副本——等于绕过写权限检查改写了一个自己只有读权限的文件，可用于覆写 setuid root 二进制或修改 `/etc/passwd` 提权。（这里的 CoW 判断逻辑，正是第二篇讲过的那套写时复制机制——它的一处竞态就是这个著名漏洞的根源。）
- **Dirty Pipe（CVE-2022-0847）**：管道缓冲区的 `PIPE_BUF_FLAG_CAN_MERGE` 标志位判断错误。通过 `splice()` 从只读文件借来的缓冲区本不该被标记为可写，却因逻辑 bug 被错误标记，导致对管道的正常写入直接落进页缓存，改写了目标文件的真实内容，全程未经任何写权限检查。相比 Dirty COW 无需凑时序竞争，更简单、更稳定。

**这正是 gVisor 与 microVM 在隔离强度上真正的优势所在。** gVisor 自研内存管理逻辑，沙箱进程不会真正触达宿主机内核的页缓存代码；microVM 更彻底——guest 拥有独立内核、独立页缓存，与宿主机页缓存在物理上是两块完全分离的内存，这类"污染宿主机页缓存"的攻击根本没有机会触达。普通容器在这一类风险上无解，只能等待内核补丁。这是三种引擎在隔离强度上真正拉开差距的地方，也印证了第一篇"隔离阶梯"的实际意义。

## 现状：三条线要分开看

把整个领域的现状讲清楚，需要把三件事分开评估：

- **工具与知识层面确实成熟了**：K8s Pod Security Admission 的 restricted 档位可在准入阶段直接拦掉 privileged、hostPID、危险 capability；OPA/Gatekeeper、Kyverno 等策略引擎、Falco 运行时监控、distroless 镜像加签名，都已是成熟方案。gVisor/microVM 这条技术线本身，就是业界承认"共享内核隔离对不可信负载不够用"之后的产物。
- **但实际部署覆盖率是另一回事**：真实安全审计中，privileged pod、挂载 docker.sock、hostPath 挂敏感目录仍然常见——多数集群未严格配置准入策略，第三方 Helm chart 的默认权限往往偏宽。最容易走的"配置性逃逸"这条路，在很多真实环境里并未被彻底堵上。
- **新漏洞没有停产**：Leaky Vessels（2024）、Dirty Pipe（2022）都不算久远。runc、containerd、内核这些共享组件持续新增功能，也持续制造新的潜在 bug，这是一场持续的攻防拉锯。

最后回到 agent 场景。传统容器安全体系的一条隐含假设是"容器里的代码是自己团队写的、review 过的"。而 agent 执行的是 LLM 生成的、可能被 prompt injection 影响的代码——这是这套体系未曾充分覆盖的新威胁模型，也是 agent sandbox 领域近年积极向 gVisor/microVM 迁移的直接动因。如果单靠容器加固就能解决问题，不会有这么强的迁移动力。
