> 本文是「沙箱底层机制」系列第 5 篇，也是收尾。前四篇沿着[隔离](/posts/sandbox-deep-dive/)、[缺页](/posts/sandbox-page-fault/)、[逃逸](/posts/sandbox-escape/)、[编排](/posts/sandbox-declarative-k8s/)四条线，把主流沙箱的机制讲了一遍。这一篇收三个"编外"话题——它们各自代表了一种前面主线之外的独立思路，恰好也是理解整个领域全貌的三块拼图。

第一篇讲隔离阶梯时，我把四种引擎归纳成四种哲学：容器"发出来拦一部分"、gVisor"发出来替你处理"、microVM"内核之上加硬件墙"、Wasm"能力从未存在过"。前三种在前几篇已经讲透，这一篇先把 Wasm 这个"最不像沙箱的沙箱"讲清楚，再补两个正交的话题：eBPF 代表的第三种安全哲学，以及 GPU 这个"内核根本管不着"的资源。

## Wasm：能力从未存在过

前三种引擎管的都是"syscall 发出去之后怎么处理"——拦截、软件限制或硬件边界。Wasm 走的是完全不同的赛道：**它压根不给你产生 syscall 的能力**。

WebAssembly 最初是给浏览器设计的二进制指令格式，目的是让 C/C++/Rust 编译出的代码能在浏览器里以接近原生的速度运行。为了能安全地跑不受信任的第三方网页代码，它天生就带内存安全和能力受限的特性。后来人们发现"安全地跑不受信任代码"这个需求在浏览器之外同样普遍——服务端插件、边缘计算、AI agent 执行沙箱都是同一类问题，于是 Wasm 被拿出浏览器语境，当成通用的沙箱隔离技术使用。下面这些机制，本质都是为浏览器安全设计的副产品，只是被复用到了新场景。

**线性内存：地址被指令集本身锁死。** Wasm 模块被运行时实例化时，会分到一块连续内存，即 linear memory。关键在于，**Wasm 代码里的地址从来不是真实机器地址**——字节码里"读地址 X"，X 只是这块缓冲区内部的偏移量，与它在宿主机 RAM 的物理位置无关。指令集本身没有能表达"缓冲区外面"的地址形式，越界访问触发 trap（一种受控错误，终止模块执行），不会真的碰到邻近内存。高效的实现方式又用到了老朋友：很多运行时用 mmap 预留一大块虚拟地址空间，只给当前实际大小开访问权限，其余标记不可访问，越界时触发硬件缺页，再由信号处理逻辑转换成 trap——这与第二篇讲的 fork/CoW 是**同一套硬件缺页机制**，只是这里借来做越界检测。

**无 syscall 能力模型。** Wasm 指令集只有算术、控制流、带边界检查的内存读写——没有任何一条指令的含义是"发起系统调用"。想跟外界打交道（读文件、联网），唯一办法是调用宿主显式提供的导入函数；宿主不提供，模块没有任何退路，因为不存在裸系统调用指令。**WASI** 就是这套标准化的导入函数集合，而且是能力模型——比如文件访问，宿主显式"预开"若干目录句柄，模块即使拼出 `../../../etc/passwd` 也会被检查是否越出授权范围。这和容器"挂最小路径"是同一思路，只是这里连"整个文件系统树"这个概念都不给。

**从设计上排除了 ROP 攻击。** 原生代码有个经典弱点：函数的返回地址和普通数据存在同一块内存（调用栈）里，缓冲区溢出可覆写返回地址，劫持执行流——这就是 ROP 攻击。Wasm 把这条路从结构上堵死了：调用栈由运行时单独维护，Wasm 指令根本访问不到；跳转是结构化的，只能跳到编译期划定的代码块边界；间接调用还会核对函数签名。这些不是"拦截"，是指令集里压根没有能表达"调用栈在哪""跳到任意偏移"的能力。

所以四档隔离可以并成一张表来看：

| 引擎 | 隔离手段发生在哪 | 本质策略 |
|------|-----------------|---------|
| 容器 | syscall 直达宿主真内核 | 软件层限制，弱 |
| gVisor | syscall 被拦截转给假内核 | 拦截——发出后截胡 |
| microVM | syscall 直达 guest 内核，其上有 EPT 硬件边界 | 硬件边界——内核之上圈边界 |
| Wasm | 指令集本身没有 syscall、越界寻址、调用栈访问的能力 | 能力缺失——相关能力从未存在 |

顺带纠正一个易错点：Wasm 的安全**不是**靠运行时检查兜底的。边界检查（mmap guard page）只是高效的实现手段，即使它有 bug，调用栈依然结构上不可达——这是两道独立的保障，不是同一道防线检查两次。

## eBPF：第三种哲学——验证后信任

Wasm 模块执行前要经过类型验证，这引出了一个和它同源的思路：eBPF。

eBPF 允许你把自己的代码加载进内核里跑（用于观测、网络、安全），但内核显然不能让任意代码随便进来。它的做法是：加载前，内核**验证器**做静态分析——遍历所有可能的执行路径而不实际运行——证明程序安全，检查内存访问是否越界、循环是否可证明有限、是否只调用批准过的辅助函数。

其中循环的处理很能说明设计取舍。最初的做法是**直接禁止一切循环**（任何反向跳转都拒绝），由此彻底绕开了理论上不可判定的停机问题——不是"判断不出来所以放行"，而是"压根不允许写这种可能判断不出来的东西"。较新的内核允许有界循环，但要求循环次数在编译期可证明有明确上限。

这就是**第三种安全哲学**。microVM 是"不管代码安不安全，先造一堵硬件墙圈住它"（隔离—遏制）；eBPF 是"先证明代码安全，再让它带着完整内核访问权限跑"（验证—信任）。为什么 eBPF 只能走验证这条路？因为它的目的就是要看到内核内部的真实状态，把它圈起来反而违背了初衷。Wasm 的执行前类型验证与它属于同一大类——都不靠运行时拦截，而是在设计和验证阶段就排除整类攻击的可能。

## GPU：内核根本管不着的资源

最后一个话题和隔离无关，属于资源调度，但它揭示了一个前面反复出现的假设的边界。

第一篇讲资源时说，CPU 和内存能被 cgroup 精细限额。**根源在于这两种资源本来就由内核直接管理**：CPU 调度器本就要决定这一刻跑哪个进程，顺手就能按 cgroup 切时间片；内存本就按页由内核管理，顺手就能按页记账。cgroup 只是插进内核本来就在走的代码路径。

GPU 不一样。**它是一块独立的协处理器，内核并不直接管理它的执行**，而是通过驱动（CUDA）访问。所以 K8s 申请 `nvidia.com/gpu: 1`，本质和 devices cgroup 管的是同一件事——能不能**打开**这个设备节点，是个二选一的开关（允许/拒绝），不是"允许用一半"这种精细限额。显存由 GPU 自己的驱动和固件管理，内核没有天然控制权，这就是 GPU 切不细的架构根源。

由此派生出几个和 CPU/内存直觉相悖的行为：

- **共享只有两条路。** 时间片（软件层，只切算力不切显存——共享同一块卡的进程用同一片显存池，一方占满会挤爆其他人）或 MIG（硬件层，A100/H100 等新卡把一张卡切成最多 7 个真正独立的实例，但每份只拿到一个分数的算力）。这又是"硬件边界更彻底但要求特定硬件、代价更高"的老套路，和 EPT vs cgroup 同构。
- **该不该切，取决于负载大小，方向容易记反。** 负载小、用不满整卡，才最该考虑共享（避免拿一整张昂贵的卡跑一个用不满的小任务）；负载大、要吃满整卡，反而不该切。
- **显存耗尽既不排队也不 OOM Kill。** 显存分配失败时，调用直接返回错误码——没有内核层面"选个受害者杀掉"的机制，也不会自动排队。多数 ML 框架会在应用层抛异常，未捕获就进程崩溃，从 K8s 视角看只是一次普通的容器崩溃，走通用重启策略，不是 GPU 专属逻辑。

至于调度，需要 gang scheduling 这类能力——分布式训练的 N 个 worker 必须同时到位才能开工，K8s 默认调度器一个个独立调度，只有部分抢到资源就会空等甚至互相卡死。解决它的 **Volcano** 不是 K8s 自带的，而是需要单独部署的 CNCF 项目，用 PodGroup 这个 CRD 声明"最少凑齐几个"，够就全调、不够就一个都不调。有意思的是，Volcano 本身正是第四篇讲的"自定义 CRD + 自己写调度器"这套模式的真实产品——别人已经写好、打包成可安装插件。

还有一个贯穿始终的观察在这里再次成立：**K8s 调度器并不理解负载**。它只比对 Pod 声明要的资源和节点上报的可用资源对不对得上，信任声明的数字，不验证是否真的够用。声明少于实际需要，调度器认为"资源够了"，运行时才会因显存不足失败——"够不够"的真实判断责任在人，不在调度器。这和第四篇里"Service 假设副本可互换"是同一类提醒：**声明式系统只对账你声明的东西，它不替你理解你真正想要什么。**

## 系列回顾

五篇写完，回头看，整个沙箱领域其实收敛在几个反复出现的观念上：

- **隔离的本质是四种哲学，不是四个档次**（拦一部分 / 替你处理 / 加硬件墙 / 能力从未存在），eBPF 又补了第五种"验证后信任"。
- **一个缺页信号被挪用了四次**——CoW、快照懒加载、Wasm 边界检查、EPT 按需映射。优秀的系统设计常常不是发明新机制，而是给旧信号找新用途。
- **隔离越强，代价越具体**——逃逸案例证明容器躲不掉内核漏洞，而 microVM 的独立内核正是它的护城河，代价是 /dev/kvm、GPU 缺口和运维复杂度。
- **管理从"改一次"走向"持续对账"**——reconcile loop 是 K8s 的世界观内核，Controller、Service、Volcano 都是它的投影。
- **声明式系统不替你理解意图**——它只对账你声明的东西。

而对做 AI Infra 的人，最实际的一条结论仍是第一篇那句：**引擎在快速商品化，真正的长期价值在 facade 层**——把易变的引擎藏在一个稳定的抽象后面。这套"沿着底层机制提问、而非对着参数表比较"的方法，比任何一个具体的引擎结论都更耐用。
